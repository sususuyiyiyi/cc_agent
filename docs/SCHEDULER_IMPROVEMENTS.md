# 调度器改进说明

## 概述

针对 2026-03-07 早上 08:00 新闻播报未按时发送的问题，我们对调度器进行了全面改进，增强了日志记录、异常通知和健康检查机制。

## 改进内容

### 1. 增强调度器日志记录

**新增功能：**
- 详细记录每次任务执行的时间、结果和错误信息
- 保存任务执行历史（成功/失败次数）
- 记录调度器启动、停止等关键事件

**日志文件位置：**
- 执行日志：`logs/scheduler_execution.log`
- 健康检查数据：`logs/scheduler_health.json`

**日志格式示例：**
```
[2026-03-07 15:32:46] [INFO] ✅ 任务已添加: news_su (每天 08:00 Asia/Shanghai)
[2026-03-07 15:32:46] [INFO] ✅ 调度器已启动，等待任务执行...
[2026-03-07 08:00:05] [INFO] 任务 news_su ✅ 成功 - 2026-03-07 08:00:05
```

### 2. 任务异常通知

**新增功能：**
- 任务执行失败时自动发送飞书通知
- 通知包含任务名称、错误信息和执行时间
- 区分普通错误和紧急错误

**通知内容示例：**
```
🚨 任务执行失败通知

任务: 每日新闻
错误: API请求超时
时间: 2026-03-07 08:00:15
```

### 3. 健康检查机制

**新增功能：**
- 每小时自动检查调度器健康状态
- 检测任务丢失（如 news_su, wellness_su, review_su）
- 检测任务长时间未执行（超过 36 小时）
- 健康检查异常时发送飞书通知

**健康检查项目：**
- ✅ 所有预期任务是否已加载
- ✅ 调度器进程是否运行
- ✅ 任务是否在正常执行周期内

**通知内容示例：**
```
🚨 调度器健康检查警告

⚠️ 健康检查警告：发现丢失的任务: news_su, wellness_su
```

### 4. 改进任务调度配置

**新增参数：**
- `misfire_grace_time=300`：5 分钟的错过执行宽容时间
- `coalesce=True`：合并错过的执行
- `max_instances=1`：防止重复执行

**作用：**
- 如果系统短暂不可用，错过执行的任务会尽快补上
- 防止任务重复执行导致的资源冲突

**查看执行日志：**
```bash

```

**查看健康检查数据：**
```bash

```

**查看标准输出日志：**
```bash

```

### 重启调度器服务

```bash

```

### 停止调度器服务

```bash

```

## 监控建议

### 日常监控

1. **每天早上检查**：确认新闻播报是否发送
2. **每周检查一次**：运行健康检查脚本，查看整体状态
3. **关注飞书通知**：及时响应异常通知

### 异常处理流程

1. **收到任务失败通知**：
   - 检查日志：`tail -20 logs/scheduler_execution.log`
   - 手动触发任务：`python3 news_agent.py`
   - 如需要，重启服务：`./manage_launchd.sh restart`

2. **收到健康检查警告**：
   - 运行健康检查脚本：`./check_scheduler_health.sh`
   - 根据检查结果采取相应措施
   - 重点关注任务丢失和长时间未执行的情况

3. **发现日志异常**：
   - 检查系统资源（内存、磁盘空间）
   - 检查网络连接（如果使用外部 API）
   - 检查飞书配置是否正确

## 预防措施总结

| 功能 | 作用 | 触发条件 |
|------|------|----------|
| 详细日志 | 记录所有任务执行情况 | 每次任务执行 |
| 异常通知 | 通知用户任务失败 | 任务执行出错 |
| 健康检查 | 检测调度器异常 | 每小时自动执行 |
| 任务恢复 | 补上错过的任务 | 使用 misfire_grace_time |

## 常见问题

### Q: 为什么之前没有收到新闻播报？

A: 调度器内部状态异常，导致任务丢失。已通过改进健康检查机制来预防此类问题。

### Q: 如何确认调度器正常工作？

A: 运行 `./check_scheduler_health.sh` 查看健康状态，所有项目应该都是 ✅。

### Q: 会收到很多通知吗？

A: 只在异常情况下发送通知，正常运行不会打扰。

### Q: 如何关闭健康检查通知？

A: 修改 `config/config.yaml`，设置 `feishu.enabled: false`。

## 技术细节

### 调度器类型

使用 `BackgroundScheduler` 而非 `BlockingScheduler`，更适合作为守护进程运行。

### 时区设置

所有任务使用 `Asia/Shanghai` 时区，确保任务在正确的本地时间执行。

### 进程管理

由 launchd 管理进程生命周期，确保服务稳定运行和自动重启。
